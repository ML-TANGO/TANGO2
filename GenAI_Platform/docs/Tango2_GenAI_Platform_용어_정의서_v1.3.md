# 📘 Tango2 GenAI Platform 플랫폼 용어 정의서 (v1.3)

> **[읽기 전 안내]** 본 용어집은 Tango2 GenAI Platform(핵심 엔진: Jonathan)을 사용하여 AI 모델을 개발·학습·배포하는 사용자를 위한 안내서입니다. 각 용어에 기재된 자원(인스턴스, 워커 등)은 플랫폼이 설치된 쿠버네티스 클러스터 환경 기준이며, 타겟 디바이스(임베디드 기기)의 물리적 자원과는 별개로 운영됩니다.

---

## 1. 핵심 개념 및 방법론 (Core Concepts)

| 용어 | 영문명 | 정의 및 상세 설명 | 참고 |
|------|--------|-------------------|------|
| MLOps | MLOps | 데이터 수집·관리부터 모델 학습·배포·운영까지 AI 개발 전 과정(End-to-End)을 체계화·자동화하는 방법론입니다. Jonathan은 MLOps 라이프사이클을 GUI 기반으로 제공하여 비개발자도 손쉽게 AI 서비스를 구축할 수 있도록 지원합니다. | |
| LLMOps | LLMOps | MLOps 라이프사이클을 LLM(대규모 언어 모델) 개발에 맞게 확장한 방법론입니다. 파운데이션 모델 준비 → 파인튜닝 → RAG 시스템 구축 → 프롬프트 엔지니어링 → 서비스 배포 단계로 이루어집니다. | Jonathan은 파인튜닝과 플레이그라운드(서비스 배포)를 통해 LLMOps 라이프사이클 전 과정을 지원합니다. |
| 추론 | Inference | 학습이 완료된 AI 모델이 새로운 입력 데이터를 받아 예측 결과를 반환하는 과정입니다. 배포된 모델 API에 요청이 들어올 때마다 추론이 수행됩니다. | 추론 품질은 모델의 학습 수준과 추론 파라미터(온도, Top-P/K 등) 설정에 따라 달라집니다. 플레이그라운드에서 파라미터를 조정하며 추론 결과를 미리 확인할 수 있습니다. |
| 파운데이션 모델 | Foundation Model | 대규모 데이터로 사전 학습된 LLM으로, 파인튜닝의 출발점이 됩니다. Jonathan에서는 Hugging Face에서 원하는 모델을 검색·불러오거나 공유 저장소(ex. Nexus)에 저장된 모델을 불러와 학습할 수 있습니다. | 파운데이션 모델을 그대로 사용할 수도 있고, 보유한 도메인 데이터로 파인튜닝하여 성능을 최적화할 수 있습니다. 모델마다 지원하는 언어, Context Window 크기, 라이선스 조건이 다르므로 선택 시 확인이 필요합니다. |
| 체크포인트 | Checkpoint | 파인튜닝 진행 중 특정 시점의 모델 가중치를 저장한 스냅샷입니다. [커밋] 기능으로 생성되며, 각 커밋은 독립적인 체크포인트로 관리됩니다. | 원하는 커밋(체크포인트)을 선택하면 해당 시점부터 학습을 재개할 수 있어, 처음부터 다시 학습할 필요가 없습니다. 또한 체크포인트를 플레이그라운드에서 테스트할 수 있습니다. |

## 2. 인프라 및 환경 (Infrastructure & Base)

| 용어 | 영문명 | 정의 및 상세 설명 | 참고 |
|------|--------|-------------------|------|
| 워크스페이스 | Workspace | 플랫폼에서 자신이 속한 작업 공간의 단위입니다. 로그인 후 워크스페이스를 선택하여 작업을 시작하며, 사용자는 동시에 여러 워크스페이스에 참여할 수 있습니다. | 워크스페이스마다 사용 가능한 GPU 등 연산 자원과 스토리지 용량이 별도로 할당됩니다. 구성원 추가·변경이나 자원 조정이 필요하면 워크스페이스 관리자에게 요청합니다. |
| 인스턴스 | Instance | 학습 또는 배포 프로젝트에 할당하는 GPU·CPU·RAM 자원 묶음의 단위입니다. 학습·배포 생성 시 사용할 인스턴스와 개수를 선택합니다. | 여러 인스턴스를 지정하면 자동으로 분산 학습이 구성됩니다. 현재 사용 가능한 인스턴스 목록과 할당 현황은 워크스페이스 대시보드에서 확인할 수 있습니다. |
| 스토리지 | Storage | 데이터셋·모델 체크포인트 등의 파일을 저장하는 공간입니다. 워크스페이스 단위로 Main Storage(학습 결과물 등 일반 파일)와 Data Storage(데이터셋 전용)로 구분됩니다. | 스토리지 사용량은 대시보드에서 실시간으로 확인할 수 있습니다. 여러 학습 프로젝트에서 같은 데이터셋을 공유해 재활용할 수 있으며, 용량 추가가 필요한 경우 워크스페이스 관리자에게 문의합니다. |
| 도커 이미지 | Docker Image | 모델 학습 및 배포에 필요한 소프트웨어 실행 환경(딥러닝 프레임워크, Python 패키지 등)을 패키징한 컨테이너 이미지입니다. | 플랫폼에서 파인튜닝과 LLM 서비스 배포에 최적화된 기본 이미지를 지원합니다. |
| 대시보드 | Dashboard | 워크스페이스 이름·구성원·인스턴스 사용 현황·스토리지 사용량·최근 실행 기록 등 워크스페이스 상태를 한눈에 확인할 수 있는 통합 화면입니다. | 로그인 후 워크스페이스를 선택하면 대시보드로 이동합니다. 현재 진행 중인 학습·배포 현황과 남은 자원 현황을 실시간으로 파악할 수 있습니다. |

## 3. 프로젝트 및 학습 (Project & Training)

| 용어 | 영문명 | 정의 및 상세 설명 | 참고 |
|------|--------|-------------------|------|
| 프로젝트 | Project | 학습·배포·데이터 전처리/분석 등 각 작업을 독립적으로 관리하는 단위입니다. 작업 유형에 따라 학습 프로젝트, 배포 프로젝트 등을 별도로 생성하여 진행합니다. | 프로젝트 생성 시 Public(워크스페이스 내 전원 공유) 또는 Private(지정 인원만 공유)으로 접근 권한을 설정할 수 있습니다. 워크스페이스 관리자는 권한에 관계없이 모든 프로젝트에 접근할 수 있습니다. |
| 모델 | Model | 파인튜닝 등을 통해 생성된 AI 알고리즘 결과물로, 플레이그라운드를 통해 추론(서비스 배포)의 대상이 됩니다. | 플랫폼 내에서 파인튜닝한 LLM, 공유 저장소나 Hugging Face에서 불러온 파운데이션 모델이 모두 '모델'로 관리됩니다. |
| 파인튜닝 | Fine-tuning | 사전 학습된 LLM(Large Language Model)에 특정 도메인·태스크 데이터를 추가 학습시켜 성능을 최적화하는 과정입니다. LoRA/DoRA 등 PEFT 기법을 활용하여 적은 자원으로도 효율적으로 수행할 수 있습니다. | 파인튜닝 진행 중 Loss 그래프 등 학습 지표를 실시간으로 확인할 수 있습니다. [커밋]을 생성해 두면 해당 시점의 모델 상태가 저장되며, 이후 원하는 커밋 시점부터 학습을 재개할 수 있습니다. |
| 데이터셋 | Dataset | 학습 및 검증에 사용되는 데이터 묶음으로, 플랫폼 스토리지에 폴더 단위로 저장·관리됩니다. 파일 직접 업로드, 수집기(공공 API / 웹 크롤러 / 원격 서버) 등 다양한 방법으로 데이터를 추가할 수 있습니다. | 데이터셋은 워크스페이스 내 구성원 간 공유되며, 여러 학습 프로젝트에서 동일한 데이터셋을 재사용할 수 있습니다. 접근 권한은 읽기&쓰기 / 읽기 전용으로 구분 설정하여 원본 데이터 보호가 가능합니다. |
| 커밋 | Commit | LLM 파인튜닝 진행 중 중간 모델 상태(체크포인트)를 저장하는 기능입니다. 하나의 모델 프로젝트에 여러 커밋(체크포인트)이 누적되어 이력으로 관리됩니다. | 커밋된 체크포인트를 로드하여 해당 시점부터 재학습이 가능합니다. |
| 학습 지표 | Metrics | 모델 학습의 진행 상황과 품질을 판단하기 위한 수치 지표입니다. ① Loss – 모델 예측값과 정답 간 오차(낮을수록 좋음) ② Grad_norm – 학습 안정성 지표(급격한 증가 시 이상 신호) ③ Learning Rate – 현재 적용 중인 학습률. 학습 중 플랫폼의 시각화 기능을 통해 실시간 그래프로 확인할 수 있습니다. | Loss가 충분히 수렴하지 않거나 Grad_norm이 급격히 증가(폭발)하는 경우, 학습률·배치 크기·그래디언트 누적 등 하이퍼파라미터 조정을 검토합니다. |

## 4. 배포 및 운영 (Deployment & Ops)

| 용어 | 영문명 | 정의 및 상세 설명 | 참고 |
|------|--------|-------------------|------|
| 배포 (서버 배포 / 테스트 배포) | Deployment | 학습된 모델을 API 형태로 서비스화하는 과정입니다. 배포가 완료되면 제공되는 API 주소를 통해 외부 애플리케이션에서 모델의 추론 결과를 요청하고 수신할 수 있습니다. | 하나의 배포에 워커(Worker)를 여러 개 생성하면 API 요청이 자동으로 분산(로드 밸런싱) 처리됩니다. 배포 화면에서 API 주소를 확인하고 외부 시스템과 연동할 수 있습니다. |
| RESTful API | RESTful API | 플랫폼에서 학습된 모델을 외부 서비스와 연동하기 위해 제공하는 웹 표준 방식의 API입니다. 배포가 완료되면 고유한 API 주소(URL)가 생성되며, 외부 애플리케이션은 이 주소로 HTTP 요청을 보내 모델의 추론 결과를 JSON 형식으로 받아올 수 있습니다. | 배포 화면에서 API 주소, 요청 형식(JSON 키·데이터 타입), 응답 형식을 확인할 수 있습니다. |
| 플레이그라운드 | Playground | 파인튜닝한 LLM 모델을 정식 배포 전에 직접 테스트해볼 수 있는 환경입니다. 추론 파라미터(온도, Top-P/K 등)를 조정하며 질의응답 품질을 확인하고, 외부 연동에 필요한 API 정보를 미리 파악할 수 있습니다. | [배포] 버튼을 눌러 자원을 할당해야 테스트 기능이 활성화됩니다. 모니터링 탭에서 API 콜 수·응답 현황·비정상 처리 건수를 실시간으로 확인할 수 있습니다. |
| 워커 | Worker | 배포 프로젝트 내에서 실제로 API 요청을 수신하고 모델 추론을 수행하여 결과를 반환하는 실행 단위입니다. 워커를 생성하기 전에 사용할 이미지, 배포 코드 위치, GPU 수 등 워커 설정을 먼저 구성합니다. | 워커를 여러 개 생성하면 API 요청 부하가 자동으로 분산됩니다. 워커별 자원 사용량·API 콜 수·비정상 처리 건수를 개별 확인할 수 있으며, 이상 발생 시 워커 로그를 다운로드하여 원인을 파악합니다. |
| 오토스케일링 | AutoScaling | 서비스 요청량 변화에 따라 워커 수를 자동으로 조정하여 안정적인 서비스를 유지하는 기능입니다. 요청이 급증하면 워커를 자동으로 추가(스케일 아웃)하고, 요청이 줄면 불필요한 워커를 줄여(스케일 인) 자원 낭비를 방지합니다. | 배포 운영 시 트래픽 예측이 어렵거나 부하 변동이 큰 서비스에서 특히 유용합니다. 로드 밸런싱과 함께 작동하여 어느 워커도 과부하 없이 요청을 분산 처리할 수 있습니다. |
| 테스트 (시뮬레이션) | Test (Simulation) | 워커가 동작 중인 배포를 대상으로 플랫폼 UI에서 직접 값을 입력하고 API 응답을 확인하는 기능입니다. 외부 서비스와 연동하기 전 배포 코드와 모델이 정상 동작하는지 쉽게 검증할 수 있습니다. | 테스트 기능은 워커가 하나 이상 동작 중인 경우에만 사용 가능합니다. 입력 값과 출력 JSON, 추론 소요 시간 등 추가 정보를 함께 확인할 수 있어 성능 사전 점검에 유용합니다. |
| 모니터링 | Monitoring | 배포된 모델 서비스의 운영 상태를 실시간으로 확인하는 기능입니다. API 콜 수(성공/실패), 상태 코드별 분포, 응답 시간, 비정상 처리 발생 건수 등의 지표를 그래프로 시각화합니다. | 플레이그라운드 모니터링 탭에서 운영 현황을 확인할 수 있습니다. 이상 징후 발생 시 워커별 로그를 다운로드하여 분석할 수 있습니다. |
| 비정상 처리 | Exception | 오류 또는 잘못된 입력 등으로 인해 모델이 정상적인 추론 결과를 반환하지 못한 API 요청 건입니다. 모니터링 화면에서 워커별 비정상 처리 건수를 그래프로 확인할 수 있습니다. | 비정상 처리가 지속적으로 발생하는 경우 워커 로그를 다운로드하여 원인(배포 코드 오류, 자원 부족, 모델 로딩 실패 등)을 파악하고 조치합니다. |

## 🧪 부록: 세부 설정 및 파라미터 용어 정의 (학습 / 배포)

> 아래 파라미터들은 LLM 파인튜닝 및 플레이그라운드(추론) 사용 시 조정하는 주요 설정값입니다. 수치를 직접 입력하거나 슬라이더(Bar) UI로 간편하게 조정할 수 있습니다.

### 1. 모델 파인튜닝 기법 및 설정 (Fine-tuning)

| 용어 | 영문명 | 정의 및 상세 설명 | 참고 |
|------|--------|-------------------|------|
| LoRA / DoRA | LoRA / DoRA | 전체 모델 가중치를 수정하지 않고 소규모 추가 학습 모듈(Adapter)만 학습하는 경량 파인튜닝 기법(PEFT)입니다. 일반적인 전체 파인튜닝에 비해 GPU 자원을 크게 절약하면서도 효과적으로 모델 성능을 개선할 수 있습니다. | DoRA는 LoRA의 개선 버전으로 학습 안정성과 성능이 향상되었습니다. 파인튜닝 설정에서 LoRA를 적용하여 파인튜닝을 실행할 수 있으며, 추후 DoRA를 지원할 예정입니다. |
| 에포크 | Number of Epochs | 전체 학습 데이터셋을 처음부터 끝까지 한 번 학습하는 것을 1 에포크라고 하며, 지정한 횟수만큼 반복 학습을 수행합니다. Modelbase 파인튜닝 파라미터 설정에서 직접 입력 또는 Bar UI로 조정합니다. | 에포크 수가 너무 많으면 과적합(Overfitting, 학습 데이터에만 최적화되어 실제 사용 성능 저하), 너무 적으면 과소적합(Underfitting)이 발생합니다. 학습 중 Loss 그래프를 보며 적절한 수를 결정합니다. |
| 그래디언트 누적 | Gradient Accumulation | GPU 메모리가 부족하여 한 번에 큰 배치로 학습하기 어려울 때, 여러 번의 소규모 배치 학습 결과를 누적한 뒤 한꺼번에 모델을 업데이트하는 방식입니다. 실효 배치 크기 = 배치 크기 × 누적 스텝 수 | 메모리가 제한된 환경에서도 큰 배치 크기로 학습한 것과 유사한 효과를 낼 수 있습니다. 누적 스텝 수가 클수록 실제로는 더 많은 데이터를 모아서 업데이트하는 셈입니다. |
| 컷오프 길이 | Cutoff Length | 학습 시 입력 데이터(프롬프트+답변)의 최대 토큰 수를 제한하는 설정입니다. 이 길이를 초과하는 부분은 잘려나가며, 값이 클수록 더 긴 문맥을 학습할 수 있으나 GPU 메모리 소비가 증가합니다. | 사용하는 LLM이 처리할 수 있는 최대 토큰 수(Context Window) 이내로 설정해야 합니다. 예를 들어 Context Window가 4,096 토큰인 모델이라면 Cutoff Length를 4,096 이하로 설정합니다. |
| 학습률 | Learning Rate | 각 학습 스텝에서 모델을 얼마나 크게 수정할지를 결정하는 값입니다. 값이 작을수록 조금씩 안정적으로 학습되고, 클수록 빠르게 학습되지만 결과가 불안정해질 수 있습니다. | 일반적으로 1e-5 ~ 5e-4 범위를 사용하며, 웜업 스텝과 함께 설정하면 학습 초기 불안정을 줄일 수 있습니다. 학습 중 Loss가 급격히 오르내리면 학습률을 낮추는 것을 검토합니다. |
| 웜업 스텝 | Warmup Steps | 학습 초기에 학습률을 0에서 목표값까지 서서히 높여가는 구간의 스텝 수입니다. 처음부터 높은 학습률로 시작할 때 발생하는 학습 불안정을 방지합니다. | 전체 학습 스텝의 5~10% 수준으로 설정하는 것이 일반적입니다. 학습 초반 Loss가 비정상적으로 튀는 경우 웜업 스텝을 늘리는 것을 검토합니다. |

### 2. 모델 배포 및 추론 파라미터 (Inference)

| 용어 | 영문명 | 정의 및 상세 설명 | 참고 |
|------|--------|-------------------|------|
| 요청 / 응답 | Request / Response | 외부 애플리케이션이 배포된 모델 API로 보내는 입력(Request)과, 모델이 추론하여 반환하는 결과(Response)의 한 쌍입니다. 플랫폼의 배포 API는 JSON 형식으로 데이터를 주고받습니다. | 배포 화면 및 플레이그라운드에서 API 주소와 요청 JSON 포맷 예시를 직접 확인할 수 있어, 외부 서비스 연동 전 사전 검증이 가능합니다. |
| 온도 | Temperature | 답변의 일관성과 다양성을 조절하는 값입니다(범위: 0~1 또는 그 이상). 낮을수록 매번 비슷하고 안정적인 답변이 생성되고, 높을수록 다양하고 창의적인 답변이 생성됩니다. | 사실 기반 Q&A나 정형화된 답변이 필요한 경우 낮은 값(0.1~0.3), 창의적인 글쓰기·아이디어 생성에는 높은 값(0.7~1.0)을 권장합니다. |
| Top P / Top K | Top P / Top K | 모델이 다음 단어를 선택할 때 후보 범위를 제한하여 품질 낮은 답변 생성을 억제하는 필터 설정입니다. · Top P: 확률 합이 P에 도달할 때까지의 상위 후보 중에서만 선택 · Top K: 확률 상위 K개의 후보 중에서만 선택 | Top P와 Top K를 함께 사용하면 두 조건을 모두 만족하는 범위 내에서 단어가 선택됩니다. 값이 너무 작으면 답변이 단조로워질 수 있으니 온도 값과 함께 균형을 맞춰 조정합니다. |
| 반복 페널티 | Repetition Penalty | 모델이 이미 생성한 단어를 반복하지 않도록 억제하는 값입니다. 1.0은 페널티 없음(기본)이며, 값이 클수록 같은 단어·문장을 반복하기 어렵게 됩니다. | 값을 너무 높게 설정하면 문맥상 반복이 자연스러운 단어까지 억제되어 오히려 답변 품질이 낮아질 수 있으므로, 1.1~1.3 내외에서 조정하는 것을 권장합니다. |
| 최대 시퀀스 | Max Sequence | 한 번의 요청에서 모델이 생성할 수 있는 최대 출력 길이(토큰 수)입니다. 응답이 이 길이에 도달하면 생성이 중단됩니다. | 너무 짧게 설정하면 답변이 중간에 잘리고, 불필요하게 크게 설정하면 응답 시간이 길어집니다. 실제 서비스 답변 길이를 고려하여 적절한 값으로 설정합니다. |
